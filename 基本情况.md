## 可能存在问题
### 问题1：
本地定时任务在电脑关机或未开机时会错过触发时间（如 08:00），导致当天简报未生成。

解决办法：
这是本地定时任务的典型问题，解决方案是做“补偿执行”。

1. 不只按“8:00触发”，还要“开机自检”
   程序启动时检查：今天 08:00 的任务是否已执行。
   如果没执行，就立刻补跑一次，并记录 last_run_at。

2. 任务状态持久化
   把每个任务最后执行时间写到本地（如 state.json / SQLite），不要只放内存。

结论：
定时触发 + 开机补偿 + 状态持久化。

### 问题2：
是否支持在 Ubuntu 主机长期运行，以及如何把生成的 Markdown 同步到 Windows 工作电脑。

解决办法：
1. 兼容性：
   当前项目技术栈（Python + APScheduler + SQLite）支持 Ubuntu，可在 Ubuntu 主机长期运行。

2. 当前阶段策略（先实现 Windows 可用）：
   先在 Windows 本地完成可运行版本与功能验证，再迁移到 Ubuntu 做常驻服务。

3. Ubuntu 常驻运行建议（后续）：
   使用 systemd 托管 `python run.py schedule`，确保开机自启、异常重启、日志可追踪。

4. Markdown 同步到 Windows 的可行方案：
   - Git 私有仓库同步（Ubuntu push，Windows pull）
   - Syncthing 双机目录同步
   - 飞书仅做通知（发送摘要或下载链接）

结论：
先 Windows 验证，后 Ubuntu 常驻；文件同步优先 Git 或 Syncthing。

### 问题3：
需要启用真实 LLM 摘要，并指定使用 qwen-max（OpenAI 标准格式）。

解决办法：
1. 摘要模块从 fallback 升级为真实 API 调用。
2. 模型使用 `qwen-max`。
3. 接口协议使用 OpenAI 标准 Chat Completions 兼容格式。
4. 调用失败保留降级策略，避免阻塞主流程。

结论：
支持接入 qwen-max（OpenAI 兼容方式），并保留稳态降级。
